第 1 章 · 簽核中心 + 加簽功能

給 所有使用者：員工 / 主管 / admin。

1.1 簽核中心是什麼

系統提供「一個地方看完所有跨類型的待簽」入口。不用切 8 個頁面找待簽單。

1.2 怎麼進到簽核中心

1登入後直接到 Dashboard（首頁）
2上方有兩個 tab：📊 總覽 · 📋 我的待簽（badge 顯示總數）
3有待簽時系統自動跳到「我的待簽」；沒有則停在「總覽」
手機 LIFF 也有對應入口：HRHub → 簽核中心（也可從首頁卡片連結進去）

1.3 簽核中心 5 大群組

群組包含類型使用對象
🧑 人事請假 / 加班 / 出差 / 補打卡 / 報帳主管 / HR
💰 經費申請 / 核銷主管 / 財務
🟣 人事異動離職 / 留停 / 異動HR / 高階主管
🟠 排班希望休 / 換班-我同意 / 換班-我核准員工 / 店長
🔵 任務任務確認所有人

每群組旁邊有 count badge 顯示該群組的待簽數。

1.4 怎麼簽一筆單

1進「我的待簽」→ 選對應群組 → 點要簽的 row
2系統會跳轉到對應的 HR 頁面，自動開啟該單明細 Modal
3Modal 結構：
  • 左半邊：申請人資料 + 表單內容 + 附件
  • 右半邊：簽核流程時間軸（每關狀態：已簽/等待/駁回）
  • 底部：簽核操作列（永遠看得到，不用滾）
4底部三顆按鈕：
  • ✓ 核准：通過此關，推進到下一關
  • ✗ 退回：點下去展開原因 textarea，必填，送出後申請人會收 LINE 通知並可編輯重送
  • 🪶 加簽：邀請第三人協助看，自己暫不簽（詳見 1.5）

1.5 加簽功能

當你發現「這張單可能需要其他人也看一下」，可以用加簽 — 不是把責任推給別人，而是請第三人也參與。

誰能發起加簽

只有當前 step 的簽核者本人能加簽。前一關已簽完的人不能、後一關還沒輪到的人也不能。

加簽人會看到什麼

  • 立即收到 LINE flex 卡（橘色頭、🪶 圖示）：「Zoey 邀請你協助加簽 / 案件 X / 加簽原因 Y / [✓ 核准加簽] [📋 查看 / 退回]」
  • 同樣會出現在加簽人的 Dashboard 我的待簽 + LIFF 簽核中心
  • 加簽人卡片上「核准加簽」可直接從 LINE 上點，不用進 LIFF（postback 直接處理）
  • 退回的話需要進 LIFF/Web 填原因（必填）

加簽後流程

加簽人動作系統做什麼誰收到 LINE
核准加簽chain 繼續，原當前簽核者繼續簽自己這關發起人「請繼續簽核」
退回加簽整單退回，狀態變「已退回 / 已駁回」申請人 + 發起人
沒動作 / 拖太久發起人可隨時撤銷加簽人「已撤銷加簽」
注意：加簽是「前加簽」— 加簽人簽完後才回到原簽核者。加簽人不能再對別人加簽（避免無限加簽鏈）。

加簽完整流程範例

1Zoey 收到一張 ETC 儲值金費用申請（3000 元），她身為執行長要核准
2Zoey 想：「金額比平常多，請會計師看一下」→ 點 [🪶 加簽]
3選加簽人「韓德森（會計師）」+ 填原因「金額較高請看」→ 送出
4韓德森 LINE 收到加簽請求 → 點「核准加簽」
5Zoey LINE 收到「韓德森已加簽通過，請繼續您的簽核」
6Zoey 點該單 → 簽自己這關 → 流程繼續到下一關

1.6 PDF 簽呈

每張單的 detail modal 右上角有「下載簽呈」按鈕，下載正式 PDF 用於存檔 / 上呈。

簽呈格式 (台灣傳統公文)：申請人 → 第 1 關 → 加簽（如果有）→ 第 2 關 → ... → 最終核可。

1.7 加簽 vs 退回 vs 詢問

動作什麼時候用
核准單據沒問題、合於規定、可推到下一關
加簽不確定，想找專業人看（會計、法務、其他主管）
退回單據有問題，要申請人修正後重送（必填原因）
線下詢問急、單純資訊確認 — 跟申請人 LINE 對話即可，不用動單據

2.4 申請流程通用步驟

所有申請類功能（請假/加班/出差/補打卡/費用/留停/希望休）長得很像，通用步驟：

4下方列表顯示你提過的所有申請，狀態 badge 顏色：

  • 已核准 — 通過了

第 3 章 · 簽核鏈設定（chain setup）

給 Admin。背後邏輯較複雜，建議仔細看完。

3.1 什麼是簽核鏈

「簽核鏈（approval_chain）」= 一個單據從送出到完全核准要經過的幾關 + 每關該由誰簽。

例：請假申請鏈 = 第 1 關「直屬主管」→ 第 2 關「人資」→ 結束

3.2 哪些表用 chain

表category備註
leave_requests 請假HRchain-aware
overtime_requests 加班HRchain-aware
business_trips 出差HRchain-aware
clock_corrections 補打卡HRchain-aware
expenses 費用報帳HR單關（不用 chain）
expense_requests 費用申請費用申請必須掛 chain，可金額分流
expense_settle 核銷(用 settle_chain_id)同上
resignation_requests 離職HRchain-aware
leave_of_absence_requests 留停HRchain-aware
personnel_transfer_requests 異動HRchain-aware
tasks 任務確認(用 task_confirmations)多 confirmations

3.3 怎麼建一條 chain

1Sidebar → 專案流程 → 簽核設定（或 /process/settings/chains）
2點「+ 新增 chain」→ 填：
  • name：chain 名稱（如「採購簽核 5 萬以下」）
  • category：HR / 費用申請 / 採購 / 留停 ...
  • min_amount / max_amount：金額分流（無金額相關設 0 / null）
  • organization_id：所屬組織
  • is_active：勾起來才會生效
3chain 建好後，新增 steps（每一關）

3.4 每個 step 的 9 種 target_type

step 的核心是「這關該由誰簽」。系統提供 9 種 target_type：

target_type意思需要設定範例
fixed_emp固定某員工target_emp_id「永遠由韓德森簽」
fixed_role固定某 role 角色target_role_id「任何 admin 都可簽」
fixed_dept固定某部門所有人target_dept_id「會計部任一位都可簽」
applicant_dept_manager申請人的部門主管無申請人在業務部 → 業務部主管簽
applicant_store_manager申請人的門市店長無申請人在永春店 → 永春店店長簽
applicant_section_manager申請人的科別主管無類似但更細粒度
specific_dept_manager指定部門的主管target_dept_id「人資部主管簽（不管申請人在哪部）」
specific_store_manager指定門市的店長target_store_id「永春店店長簽」
specific_section_manager指定科別的主管target_section_id同 dept_manager 更細
關鍵：applicant_* 系列是「動態解」— 系統依當前申請人解出該層級的主管。specific_* 跟 fixed_* 是「靜態指定」。
常見錯誤：多關都用同一個 applicant_dept_manager → 解出來都是同一人 → 多關形同一關。建議多關時混搭不同 target_type。

3.5 金額分流

同 category 可以建多條 chain，依金額自動挑：

chainmin_amountmax_amount適用
「費用申請 5K 以下」050001 關：直屬主管
「費用申請 5K-5W」5001500002 關：主管 + 財務
「費用申請 5W 以上」50001NULL3 關：主管 + 財務 + 總經理

系統 INSERT expense_request 時自動挑符合金額的 chain。

3.6 form_chain_configs：HR 表單怎麼掛 chain

HR 5 表 + HR 異動 3 表不靠金額分流，而是靠「form_type → chain 對應」：

form_type對應
leave請假
overtime加班
trip出差
correction補打卡
resignation離職
loa留停
transfer異動

各表的「簽核設定」按鈕（每張表單頁右上）會跳到設定頁。

第 4 章 · 主管簽核作業

給 店長 / 部門主管。日常簽核作業的最佳實踐。

4.1 三種簽核入口

入口優點缺點適合
LINE 卡即時、不用開系統畫面小、退回要進 LIFF 填原因外出時、快速核准
LIFF 簽核中心手機完整視圖 + 加簽需要 LINE 開啟手機處理、批量簽
Web Dashboard大畫面、看完整明細 + PDF需電腦金額大、需細看內容

4.2 LINE 卡上的快速操作

1收到簽核卡 → 看內容
2三個 footer 按鈕：
  • ✅ 核准：直接點 → 推進到下一關
  • ❌ 退回：跳出快速回覆選項（需附證明 / 日期改 / 工時不允許 / ✏️ 自己寫）
  • 📋 明細：跳 LIFF 看完整資料

4.3 看明細決定時的判斷流程

1看申請人：是不是我的下屬 / 對的人
2看時間 / 金額：合理範圍嗎
3看原因：說明清楚嗎
4看附件（如有）：證明文件齊不齊
5看歷程：之前關卡誰簽過、簽核意見是什麼
6決定：核准 / 加簽 / 退回

4.4 退回的最佳實踐

  • 原因要具體：「請補附發票照片」優於「資料不足」
  • 建議如何修：「日期請改成上班日」優於「日期錯」
  • 避免主觀：「金額過高」要說明標準（依公司 SOP / 比照前例）

4.5 批次簽核小技巧

1Web Dashboard → 我的待簽 → 選擇對應群組
2看到一堆類似的單（如月初員工集中報帳）：先全部點開明細快速掃
3確認都 OK 後一個個 modal 內點核准（沒有真正的批次按鈕 — 簽核屬於審慎動作）

4.6 不該做的事

  • ❌ 不細看內容直接「秒簽」批量 — 出事責任在你
  • ❌ 直接幫部屬代為提單再自己簽 — 違反「自簽不允許」的設計
  • ❌ 用聊天 / 口頭核准代替系統簽核 — 沒紀錄無法追溯
  • ❌ 拖過久不簽 — 同事的薪資 / 排班可能因此卡關

第 5 章 · 排班系統

給 店長 / HR。排班是相對複雜的後台運算，這章說明設計概念與操作。

5.1 系統採用的排班制度

制度說明
月制工時以月為單位算總工時，不是週固定。一個月不超過 174h 等（依勞基法）
時段覆蓋制排班不是員工選班別，而是「店長定義時段（早班 / 中班 / 晚班 / 大夜）」員工去 cover
三級偏好員工可標記某些時段「希望 / 普通 / 不希望」，演算法盡量配合

5.2 排班相關設定（admin / 店長）

1Sidebar → HR → 排班管理 → 排班規則 / 工時設定
2每個門市可獨立設定每月時段（store_time_slots）
3每個門市可獨立設定班別（shift_definitions）：班名 / 起訖時間 / 顏色
重要：shift_definitions 跟 store_time_slots 必須帶 store_id + year_month，跨月複製要先設定。

5.3 員工提交希望休

1員工：LIFF HRHub → 各類申請 → 排休申請
2選日期 → 填原因 → 送出
3店長收 LINE 通知 → 核准 / 退回
4核准後該日期會被排班演算法視為「請避開」

5.4 跑排班演算法

1Web → HR → 排班
2選月份 + 門市 + 班別範圍
3點「LLM 排班 / 自動排班」
4系統會考慮：員工偏好 / 希望休 / 月工時上限 / 加班上限 / 排班規則 / 正職優先
5產生 draft，店長可手動微調 → 確認後發佈

5.5 換班流程

1員工 A：LIFF → 換班申請 → 選對方 B + 日期
2B 收 LINE → 同意 or 拒絕
3B 同意 → 進入「待主管核准」階段，店長收通知
4店長核准 → 系統實際交換 schedules 表的班次 + 工時欄位
5A / B 都收到「換班通過」LINE 通知

5.6 排班月休邏輯

  • 休假天數按月累計（不要拆週分配）
  • 純月制目標：每月若需休 N 天則整月內配置 N 天
  • 正職員工先排，PT 補空檔

5.7 常見排班問題

問題原因處理
演算法跑出來人力不夠時段定義太多 / 員工太少調整時段、增聘、減少時段
某員工工時破表偏好過少 / 班別設計問題手動微調、確認班別覆蓋率
希望休全被忽略還沒核准 / 跨月設定錯確認希望休狀態 + year_month

第 6 章 · 薪資作業 SOP

給 HR / Payroll。算錯薪資後果嚴重，這章是鐵則 + SOP。

6.1 月底薪資作業流程

1確認當月出勤資料齊全（打卡 / 補打卡 / 請假 / 加班）
2跑「批次計薪」（salary.compute 權限）
3對照各員工的明細（底薪 / 加班 / 津貼 / 扣項）
4跟廠商（會計師事務所）的計算 PDF 比對
5差異排查 → 調整 → 重算
6產出薪資單 + 發放

6.2 薪資計算鐵則（不要違背）

  • 所得稅不代扣：系統 NOT 自動扣所得稅。員工自行處理年度報稅。
  • 月薪不算國定假日：月薪制員工的國定假日已包含在底薪內，不另外算加班費
  • 投保用 base + role cap：投保級距上限 45,800（依勞動部）
  • PT 固定 11,100 / 29,500：PT 兼職員工特殊投保算法

6.3 加班費三桶

桶倍率條件
第 1 桶1.34平日加班前 2 小時
第 2 桶1.67平日加班 2 小時後
第 3 桶2.67休息日 / 國定假日加班

6.4 勞健保級距

1Sidebar → HR → 薪酬與福利 → 健保級距監控
2每月看實領 vs 投保級距是否一致
3不一致 → 需調整級距（依勞健保局規定有調整時機）

6.5 永春案例（PT 投保特例）

永春 PT 投保用固定值 11,100 / 29,500，不照一般員工級距表算。這是廠商特殊規則。

6.6 常見薪資問題

問題排查方向
某員工薪資少算 / 多算確認出勤資料 / 假單 / 加班 / 補打卡是否齊全；確認 employee_salary_structures 設定
批次計薪報錯看哪幾位錯 → 個別檢查；看 audit log
勞健保級距異常確認 insurance_grade_monitor；可能級距需手動調整
所得稅相關詢問系統不代扣，請員工自行處理；HR 不該回答稅務問題

第 8 章 · Admin 緊急排錯手冊

給 admin / 開發者。系統出狀況時的應急處理。

8.1 常見錯誤類型

A. 系統閃退 / 白畫面（React 錯誤）

典型訊息：「ReferenceError: XXX is not defined」

1F12 開瀏覽器 console，看具體錯誤訊息 + stacktrace
2多半是 import 漏（重構切檔忘記補）
3找到對應檔案，加 import 即可

已發生案例：Award is not defined（Sidebar）/ SOURCE_BADGE is not defined（EmployeePermissions）

B. Migration drift（DB schema 跟 code 對不上）

典型錯誤：「column does not exist」「relation not found」

1確認哪張表哪個欄位
2到 Supabase Studio 查 information_schema.columns 確認真實 schema
3如果 schema 是新版但 migration file 是舊版 → 補 migration / 修 code
4如果是老闆直接 Studio 改 → 更新 migration file 保持同步

C. 簽核流程卡住

單據卡在某關，沒人能簽。

1查該單 approval_chain_id + current_step
2查對應 chain_step 的 target_type 跟解出來的 approver
3常見：applicant 的 supervisor / store_manager 沒設 → 解不出簽核人
4修：到員工資料補上 supervisor_id / 門市設 manager_id

D. 加簽幽靈單

workflow_instances + tasks 跟 expense_request 不同步，LINE 推「今日到期提醒」吵人。

1確認該 workflow_instance / tasks 對應的 expense_request 真實狀態
2如果原單已結案，把 wi + tasks 標已完成（不動原單）
3SQL：UPDATE workflow_instances SET status='已完成' WHERE id IN (...)

E. LINE 推不出去

檢查項方法
該員工有沒有 LINE 綁定SELECT * FROM employee_line_accounts WHERE employee_id = X
line-push Edge Function 健康看 Supabase functions log
LINE channel token 過期看 line_channels 表 access_token + 重新整理
net.http_post 失敗查 pg_net 設定 + extensions

8.2 SQL 緊急修法（要小心！）

鐵則：
  • 所有 SQL update 先 BEGIN; ... 觀察結果後再 COMMIT
  • 影響範圍要先 SELECT 列出來看清楚
  • DELETE / DROP 等不可逆動作絕對先列影響 → 確認 → 才動手
  • 2026-05-13 慘案：誤解「加班就洪伯嘉就好」結果刪錯方向，22 筆非洪伯嘉資料蒸發

指令功能範例

9.2 卡片按鈕（postback）

LINE 收到的 flex 卡上有按鈕，點下去會做：

按鈕動作
✅ 核准對該單核准（不用進 LIFF）
❌ 退回跳 quick reply 選常用原因，或進 LIFF 自己寫
📋 明細跳 LIFF 看完整資料
✏️ 編輯重送（被退回後）跳 LIFF 編輯重新送審
🪶 核准加簽對加簽請求直接核准（不用進 LIFF）

9.3 LINE 通知卡片類型

卡片什麼時候收到
簽核請求卡輪到你簽某張單
加簽請求卡有人邀請你協助加簽
已核准通知自己的單通過了
已退回通知自己的單被退回（含原因 + 編輯重送鈕）
到期提醒排程提醒當日待簽（每天早上推）
任務確認卡任務 chain 輪到你確認

9.4 LINE 卡片不見了？

LINE 客戶端有快取，如果系統 deploy 後卡片格式變了，可能要重新觸發指令才能看到新卡。

1輸入指令重新觸發（例：/我的待簽）
2或請發出單據的人重新觸發那個事件

9.5 LINE 收不到通知

檢查怎麼做
LINE 有沒有封鎖官方帳號檢查好友列表
有沒有解除綁定輸入 /我的單 看會不會回
LINE 通知總開關LINE app → 設定 → 通知
請 HR / Admin 重新綁後台檢查 employee_line_accounts
