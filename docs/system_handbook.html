第 1 章 · 簽核中心 + 加簽功能

給 所有使用者：員工 / 主管 / admin。

1.1 簽核中心是什麼

系統提供「一個地方看完所有跨類型的待簽」入口。不用切 8 個頁面找待簽單。

1.2 怎麼進到簽核中心

1登入後直接到 Dashboard（首頁）
2上方有兩個 tab：📊 總覽 · 📋 我的待簽（badge 顯示總數）
3有待簽時系統自動跳到「我的待簽」；沒有則停在「總覽」
手機 LIFF 也有對應入口：HRHub → 簽核中心（也可從首頁卡片連結進去）

1.3 簽核中心 5 大群組

群組包含類型使用對象
🧑 人事請假 / 加班 / 出差 / 補打卡 / 報帳主管 / HR
💰 經費申請 / 核銷主管 / 財務
🟣 人事異動離職 / 留停 / 異動HR / 高階主管
🟠 排班希望休 / 換班-我同意 / 換班-我核准員工 / 店長
🔵 任務任務確認所有人

每群組旁邊有 count badge 顯示該群組的待簽數。

1.4 怎麼簽一筆單

1進「我的待簽」→ 選對應群組 → 點要簽的 row
2系統會跳轉到對應的 HR 頁面，自動開啟該單明細 Modal
3Modal 結構：
  • 左半邊：申請人資料 + 表單內容 + 附件
  • 右半邊：簽核流程時間軸（每關狀態：已簽/等待/駁回）
  • 底部：簽核操作列（永遠看得到，不用滾）
4底部三顆按鈕：
  • ✓ 核准：通過此關，推進到下一關
  • ✗ 退回：點下去展開原因 textarea，必填，送出後申請人會收 LINE 通知並可編輯重送
  • 🪶 加簽：邀請第三人協助看，自己暫不簽（詳見 1.5）

1.5 加簽功能

當你發現「這張單可能需要其他人也看一下」，可以用加簽 — 不是把責任推給別人，而是請第三人也參與。

誰能發起加簽

只有當前 step 的簽核者本人能加簽。前一關已簽完的人不能、後一關還沒輪到的人也不能。

加簽人會看到什麼

  • 立即收到 LINE flex 卡（橘色頭、🪶 圖示）：「Zoey 邀請你協助加簽 / 案件 X / 加簽原因 Y / [✓ 核准加簽] [📋 查看 / 退回]」
  • 同樣會出現在加簽人的 Dashboard 我的待簽 + LIFF 簽核中心
  • 加簽人卡片上「核准加簽」可直接從 LINE 上點，不用進 LIFF（postback 直接處理）
  • 退回的話需要進 LIFF/Web 填原因（必填）

加簽後流程

加簽人動作系統做什麼誰收到 LINE
核准加簽chain 繼續，原當前簽核者繼續簽自己這關發起人「請繼續簽核」
退回加簽整單退回，狀態變「已退回 / 已駁回」申請人 + 發起人
沒動作 / 拖太久發起人可隨時撤銷加簽人「已撤銷加簽」
注意：加簽是「前加簽」— 加簽人簽完後才回到原簽核者。加簽人不能再對別人加簽（避免無限加簽鏈）。

加簽完整流程範例

1Zoey 收到一張 ETC 儲值金費用申請（3000 元），她身為執行長要核准
2Zoey 想：「金額比平常多，請會計師看一下」→ 點 [🪶 加簽]
3選加簽人「韓德森（會計師）」+ 填原因「金額較高請看」→ 送出
4韓德森 LINE 收到加簽請求 → 點「核准加簽」
5Zoey LINE 收到「韓德森已加簽通過，請繼續您的簽核」
6Zoey 點該單 → 簽自己這關 → 流程繼續到下一關

1.6 PDF 簽呈

每張單的 detail modal 右上角有「下載簽呈」按鈕，下載正式 PDF 用於存檔 / 上呈。

簽呈格式 (台灣傳統公文)：申請人 → 第 1 關 → 加簽（如果有）→ 第 2 關 → ... → 最終核可。

1.7 加簽 vs 退回 vs 詢問

動作什麼時候用
核准單據沒問題、合於規定、可推到下一關
加簽不確定，想找專業人看（會計、法務、其他主管）
退回單據有問題，要申請人修正後重送（必填原因）
線下詢問急、單純資訊確認 — 跟申請人 LINE 對話即可，不用動單據

第 3 章 · 簽核鏈設定（chain setup）

給 Admin。背後邏輯較複雜，建議仔細看完。

3.1 什麼是簽核鏈

「簽核鏈（approval_chain）」= 一個單據從送出到完全核准要經過的幾關 + 每關該由誰簽。

例：請假申請鏈 = 第 1 關「直屬主管」→ 第 2 關「人資」→ 結束

3.2 哪些表用 chain

表category備註
leave_requests 請假HRchain-aware
overtime_requests 加班HRchain-aware
business_trips 出差HRchain-aware
clock_corrections 補打卡HRchain-aware
expenses 費用報帳HR單關（不用 chain）
expense_requests 費用申請費用申請必須掛 chain，可金額分流
expense_settle 核銷(用 settle_chain_id)同上
resignation_requests 離職HRchain-aware
leave_of_absence_requests 留停HRchain-aware
personnel_transfer_requests 異動HRchain-aware
tasks 任務確認(用 task_confirmations)多 confirmations

3.3 怎麼建一條 chain

1Sidebar → 專案流程 → 簽核設定（或 /process/settings/chains）
2點「+ 新增 chain」→ 填：
  • name：chain 名稱（如「採購簽核 5 萬以下」）
  • category：HR / 費用申請 / 採購 / 留停 ...
  • min_amount / max_amount：金額分流（無金額相關設 0 / null）
  • organization_id：所屬組織
  • is_active：勾起來才會生效
3chain 建好後，新增 steps（每一關）

3.4 每個 step 的 9 種 target_type

step 的核心是「這關該由誰簽」。系統提供 9 種 target_type：

target_type意思需要設定範例
fixed_emp固定某員工target_emp_id「永遠由韓德森簽」
fixed_role固定某 role 角色target_role_id「任何 admin 都可簽」
fixed_dept固定某部門所有人target_dept_id「會計部任一位都可簽」
applicant_dept_manager申請人的部門主管無申請人在業務部 → 業務部主管簽
applicant_store_manager申請人的門市店長無申請人在永春店 → 永春店店長簽
applicant_section_manager申請人的科別主管無類似但更細粒度
specific_dept_manager指定部門的主管target_dept_id「人資部主管簽（不管申請人在哪部）」
specific_store_manager指定門市的店長target_store_id「永春店店長簽」
specific_section_manager指定科別的主管target_section_id同 dept_manager 更細
關鍵：applicant_* 系列是「動態解」— 系統依當前申請人解出該層級的主管。specific_* 跟 fixed_* 是「靜態指定」。
常見錯誤：多關都用同一個 applicant_dept_manager → 解出來都是同一人 → 多關形同一關。建議多關時混搭不同 target_type。

3.5 金額分流

同 category 可以建多條 chain，依金額自動挑：

chainmin_amountmax_amount適用

第 4 章 · 主管簽核作業

給 店長 / 部門主管。日常簽核作業的最佳實踐。

4.1 三種簽核入口

入口優點缺點適合
LINE 卡即時、不用開系統畫面小、退回要進 LIFF 填原因外出時、快速核准
LIFF 簽核中心手機完整視圖 + 加簽需要 LINE 開啟手機處理、批量簽
Web Dashboard大畫面、看完整明細 + PDF需電腦金額大、需細看內容

4.2 LINE 卡上的快速操作

1收到簽核卡 → 看內容
2三個 footer 按鈕：
  • ✅ 核准：直接點 → 推進到下一關
  • ❌ 退回：跳出快速回覆選項（需附證明 / 日期改 / 工時不允許 / ✏️ 自己寫）
  • 📋 明細：跳 LIFF 看完整資料

4.3 看明細決定時的判斷流程

1看申請人：是不是我的下屬 / 對的人
2看時間 / 金額：合理範圍嗎
3看原因：說明清楚嗎
4看附件（如有）：證明文件齊不齊
5看歷程：之前關卡誰簽過、簽核意見是什麼
6決定：核准 / 加簽 / 退回

4.4 退回的最佳實踐

  • 原因要具體：「請補附發票照片」優於「資料不足」
  • 建議如何修：「日期請改成上班日」優於「日期錯」
  • 避免主觀：「金額過高」要說明標準（依公司 SOP / 比照前例）

4.5 批次簽核小技巧

1Web Dashboard → 我的待簽 → 選擇對應群組
2看到一堆類似的單（如月初員工集中報帳）：先全部點開明細快速掃
3確認都 OK 後一個個 modal 內點核准（沒有真正的批次按鈕 — 簽核屬於審慎動作）

4.6 不該做的事

  • ❌ 不細看內容直接「秒簽」批量 — 出事責任在你
  • ❌ 直接幫部屬代為提單再自己簽 — 違反「自簽不允許」的設計
  • ❌ 用聊天 / 口頭核准代替系統簽核 — 沒紀錄無法追溯
  • ❌ 拖過久不簽 — 同事的薪資 / 排班可能因此卡關

第 5 章 · 排班系統

給 店長 / HR。排班是相對複雜的後台運算，這章是詳細版，含演算法 v5 內部邏輯。

5.1 系統採用的排班制度

5.1.1 月制工時（非週制）

以月為單位計算總工時上限，不是每週固定。

  • 勞基法基準：每月正常工時 174 小時內
  • 例外加班：每月加班不超過 46 小時（一般行業）
  • 系統會在 store_settings 內讓店長設「每月工時上限」、「加班上限」
為什麼選月制：用週制控制工時時邊界 case 太多（連續週交界、月底切點），月制計算精準到目標天數，演算法簡單可靠（2026-04-21 從週制改為純月制）。

5.1.2 時段覆蓋制（非員工選班別）

排班邏輯不是「員工自己選想上的班別」，而是：

  1. 店長先定義店時段（store_time_slots）— 一天分早 / 中 / 晚 / 大夜等時段
  2. 每個時段定義需求人數（早班需 2 人 / 晚班需 3 人等）
  3. 演算法把員工分配進時段覆蓋，確保每個時段都有足夠人手

5.1.3 班別 vs 時段

概念是什麼store_id + year_month 必填
shift_definitions班別（標準班次）：早班 09-17、晚班 17-23 等。員工被分配「班別」。✅ 必填
store_time_slots店時段（營業覆蓋）：早 10-14、中 14-18、晚 18-22 等。系統檢查覆蓋率。✅ 必填
坑：shift_definitions 跟 store_time_slots 沒帶 store_id 會混到其他門市的班別。每個月份要先建好對應月的設定。

5.1.4 三級偏好制（演算法分數）

員工對每個班別 / 時段可表達偏好（在 employee_shift_preferences）：

偏好icon演算法分數意義
希望⭐+20強烈想上這時段
可以✅+8普通可以
不行❌blocked排不進去（硬約束）
分數差別：演算法把所有可能組合算總分，分數最高的方案勝出。⭐ 跟 ✅ 差距大（+20 vs +8）讓「強烈偏好」真的有差。

5.2 排班相關設定步驟（admin / 店長）

5.2.1 設定店時段

1Sidebar → HR → 排班管理 → 排班規則 / 工時設定 → 選門市
2「店時段設定」tab → 新增時段：時間範圍 + 需求人數
3跨月需重複設定（系統會自動帶上 year_month）

5.2.2 設定班別

1「班別管理」tab → 新增班別：班名 / 起訖時間 / 顏色 / 適用員工類型（全職 / PT / 全部）
2同個店可有多個班別組合

5.2.3 設定每月工時 / 月休

1「工時設定」tab
2填：每月正常工時 / 加班上限 / 月休天數（正職）/ 月休天數（PT）
3系統會以這個值為演算法目標

5.2.4 員工偏好（員工自己設或店長代設）

1員工：LIFF → 個人偏好設定（或店長後台代設）
2對每個班別 / 時段標 ⭐ / ✅ / ❌
3偏好會直接餵進演算法

5.3 員工提交希望休（off_request）

1員工：LIFF HRHub → 各類申請 → 排休申請
2選日期 → 填原因 → 送出
3店長收 LINE 通知 → 核准 / 退回（每月通常有截止日，截止後不收）
4核准後該日期會被演算法視為「請避開」
員工也可在 Web 看希望休狀態（HRHub → 各類申請 → 排休申請）。如被退回，原因會顯示。

5.4 排班演算法 v5（核心邏輯）

5.4.1 整體流程

月排班開始
  ├─ 拆成週（4 週變形月制）
  └─ 每週依序：
     1. 標記休息日（依員工已核准的希望休 + 月休目標）
     2. Pass 1：偏好班別分配（⭐ 優先 → ✅ 次之）
     3. Pass 2：剩餘人力評分填滿（覆蓋率優先）
     4. Step 3a：混合補班（標準班別不夠時動態產生非標準班次）
     5. Step 3b：開店 / 關店人員修正（早班一定要有開店人）
     6. Step 3c：跨門市借調（本店人力不足時從 additional_stores 借）
     7. 驗證 + 統計（hour balance、覆蓋率、違規檢查）
     8. [可選] LLM retry（有違規時用 Gemini 建議微調）
最終校正：超標的月休轉回上班

5.4.2 三級偏好分數計算

對每一個「員工 × 時段 / 班別」組合算分：

  • 基礎分數 = 員工偏好（⭐+20 / ✅+8 / ❌blocked）
  • + 加成：員工這月還欠工時 → 加分
  • + 加成：員工偏好覆蓋率高 → 加分
  • - 懲罰：跨門市借調 -20 分（盡量本店人）
  • - 懲罰：連續上班過久 → 扣分
  • - 硬約束（不會分配）：希望休、月休目標已達、月工時上限已到

5.4.3 月休純月制邏輯（2026-04-21 大改）

  • 店長設「月休 N 天」（如正職 10 天 / PT 14 天）
  • 演算法在月內均勻配置，不再用週分配
  • Step 1c：正職的硬性月休先排（這是硬約束）
  • Phase 3：不給正職額外休假，只補 PT 月休彈性
  • 最終校正：演算法超標時，把多排的休假轉回上班（保證精確命中目標）
過去 bug：用週分配（ceil/floor/max）控制月休 → 邊界 case 11/12/13 天不穩定。改純月累計後精確到目標。

5.4.4 LLM retry（演算法 v5 新增）

跑完演算法後驗證，若有違規（如某員工工時超標、某時段沒覆蓋）：

  1. 送 Gemini：「現有方案有 X 違規，建議調整」
  2. Gemini 回保守建議（盡量小幅調整不大改）
  3. 系統 retry 演算法 with 新限制
  4. 最多 retry 3 次，仍違規時呈現給店長手動處理

5.4.5 跨門市借調

多店連鎖時，本店人力不足可從「附屬門市」借人：

  • 需在 employees.additional_stores 設定「員工跨支援的門市」
  • 跨店上班 -20 分懲罰（盡量本店）
  • 跨店上班結果會出現在借調員工的本月班表 + 兩店都看得到

5.4.6 歷史模式分析（analyzeHistoricalPatterns）

跑演算法前，系統會先分析過去 4-8 週的排班：

  • 找出固定模式（如周一固定 A 開店、週末固定 B 顧）
  • 提供「歷史傾向」加分到當月演算法
  • 讓排班結果保持連續性，不會每月都跑出截然不同的安排

5.5 排班結果處理

5.5.1 Draft → Publish

1跑演算法後系統產出 draft（草稿）
2店長手動微調（拖拉某員工換時段）
3確認後點「發佈」→ 寫入 schedules 表，員工才看得到

5.5.2 publishChecklist 檢查項

發佈前系統會列出檢查項：

檢查項意義
覆蓋率 100%每個店時段都有足夠人力
無人工時超標沒有員工月工時超過上限
無違反勞基法連續工時 / 休息時數合於法規
希望休全部尊重核准的希望休都被避開
月休天數精確每人月休天數 = 設定目標

5.6 換班流程（shift_swap）

1員工 A：LIFF → 換班申請 → 選對方 B + 日期
2B 收 LINE 「[同意] [拒絕]」
3B 同意 → 狀態變「待主管核准」，店長收通知
4店長核准 → 系統實際交換 schedules 表的：
  • shift（班次名）
  • actual_start / actual_end（上下班時間）
  • actual_hours（工時）
5A / B 都收 LINE「換班通過」通知

5.7 常見排班問題

問題原因處理

6.2 薪資組成公式

6.2.1 月薪正職

月薪 = 底薪(base)
     + 職務津貼(role_allowance)
     + 其他津貼（伙食、交通、全勤等）
     + 加班費（依加班桶分別計算）
     + 績效獎金
     - 請假扣款（依日 / 時計）
     - 投保自付（勞保 + 健保）
     - 法扣（如有）
     - 預支 / 借款扣款（如有）
     = 實領

6.2.2 PT 兼職

PT 薪資 = 工作時數 × 時薪
       + 加班費（同月薪規則）
       + 國定假日 / 例假日加成（×2 詳見 6.4.3）
       - 投保自付（PT 特殊 11,100 / 29,500）
       - 法扣（如有）
       = 實領

6.2.3 hourly rate 公式（重要）

🔴 鐵則：加班費的 hourly rate = base / 30 / 8，不是全薪 / 240。
  • 正職月薪 30,000 → hourly = 30000 / 30 / 8 = 125
  • 加班 1 小時平日 1.34 倍 → 125 × 1.34 = 167.5
  • 用全薪 (含津貼) 算 hourly 會多算 → 違反勞基法

6.3 加班費規則（A/B/C 三類）

6.3.1 A 類：平日延長工時（勞基法 §24）

時段倍率說明
前 2 小時1.34正常工時(8h) 外延長前 2 小時
2 小時後1.67延長超過 2 小時的部分

例：平日上班 8 小時後加班 3 小時 → 前 2h ×1.34 + 第 3h ×1.67

6.3.2 B 類：休息日加班（勞基法 §24-1）

休息日 = 一般指週六（公司可調，看「彈性出勤」設定）

時段倍率說明
前 2 小時1.34休息日加班前 2 小時
2-8 小時1.67第 3-8 小時
8-12 小時2.67第 9-12 小時（這就是「2.67 桶」）

例：週六出勤 10 小時 → 前 2h ×1.34 + 中間 6h ×1.67 + 後 2h ×2.67

6.3.3 C 類：國定假日 / 例假日 出勤（勞基法 §39）

⚠️ 跟加班費不同： 國定假日 / 例假日 出勤本身就是工資加倍（不是加班費的概念）。即便只上 1 小時也是這天工資全額再給一倍。
對象怎麼算
月薪正職底薪已含整月工資（含國定假日）→ 不另外加給
PT 兼職工時 ×1 倍時薪 + 額外 ×1 倍 holidayBonus = 共 ×2

例 1：月薪 30,000 員工在 4/4 兒童節上班 4 小時 → 不另外給（底薪已含）

例 2：PT 時薪 200 在 4/4 兒童節上班 4 小時 → 4 × 200 (工時) + 4 × 200 (holidayBonus) = 1,600

6.3.4 D 類：例假日（週日）

跟國定假日同邏輯：×2 工資加倍。一般不應出勤，必須出勤的話需先取得勞工同意。

6.3.5 ot_category 自動分類

系統依加班日期自動判 weekday / restday / holiday，存在 overtime_requests.ot_category。

category對應用哪桶
weekday週一到週五（非國定）A 類 1.34 / 1.67
restday休息日（週六）B 類 1.34 / 1.67 / 2.67
holiday國定假日 / 例假日C 類 ×2（PT 適用，月薪不加）
注意：ot_type = pay / leave，是「加班補休 vs 加班費」的選擇，不是平日/休息日分類。過去把 ot_type 誤當分類 → 全算平日倍率。

6.3.6 加班 start_time / end_time（2026-05-14 新增）

過去只記 hours，不知道實際幾點到幾點。現在補上：

  • start_time：加班起始時間（24h 制）
  • end_time：加班結束時間（可跨日）
  • 用途：稽核「平日 18:00-19:00 vs 凌晨 03:00-04:00 是否倍率不同」

6.3.7 加班補休 vs 加班費（ot_type）

ot_type意思
pay領加班費（依倍率計算）
leave換補休（hours 進 leave_balances）

申請時員工自己選。同一筆加班只能擇一。

6.4 投保金額規則

6.4.1 月薪正職

1計算 insured = base + role_allowance（基本薪 + 職務津貼）
2對照勞健保級距表找到對應級距
3上限 cap 45,800（依勞動部 2026 級距）
4勞保 / 健保用同一個值（廠商實務 — 把健保也 cap 45,800，雖違反健保法但常見）

6.4.2 PT 兼職（特殊）

險種固定值
勞保11,100
健保29,500
法規 vs 實務：法規 PT 應用「最近 3 個月平均月薪資」對齊級距，2 月跟 8 月各調一次。廠商實務常用固定值（簡化）。系統預設廠商實務 (forcePartTimeMin=true)，admin 可後台 base_insured 覆寫。

6.4.3 2026 勞保費率 12.5%

項目費率
普通事故保險費率11.5%
+ 就業保險費率1.0%
= 合計12.5%
65 歲以上（已退休再就業）11.5%（無就保）
員工自付 20%、雇主負擔 70%、政府補助 10%（一般）— 比例依勞保條例。

6.4.4 健保費率

2026 健保費率 5.17%（一般）。員工自付 30%、雇主 60%、政府 10%。

6.4.5 勞退 6%（雇主提撥）

系統 hardcode 計算：級距 × 6%（雇主全額提撥，員工不扣）。

6.4.6 admin 覆寫

如果廠商規則跟系統算出來不同（特例員工），admin 可在後台填 employee_salary_structures.base_insured，系統會優先用這個值，不再依公式算。

6.5 計薪鐵則（嚴格遵守，不要違背）

  1. 所得稅不代扣：系統 withholdTax: false。員工 5 月自行報稅。HR 不該回答稅務問題。
  2. 月薪正職國定假日不另加：月薪固定值已含整月工資，holidayBonus 只對 PT。
  3. PT 國定假日 ×2：工時 1 倍 + holidayBonus 1 倍 = 共 ×2。
  4. 投保 = base + role 對齊級距，上限 45,800。健保也 cap 45,800（廠商實務）。
  5. PT 投保固定 11,100 / 29,500（廠商實務）— 法規應用 3 個月平均，但簡化用固定值。
  6. 加班 hourly rate = base / 30 / 8（不是全薪 / 240）。
  7. 2026 勞保費率 12.5%（普通 11.5% + 就保 1%）。
  8. ot_category 才是平日/休息日分類，ot_type 是補休 vs 加班費。不要混淆。

4停用 LINE 綁定 + 使用者帳號（不要刪、保留歷史）

第 8 章 · Admin 緊急排錯手冊

給 admin / 開發者。系統出狀況時的應急處理。

8.1 常見錯誤類型

A. 系統閃退 / 白畫面（React 錯誤）

典型訊息：「ReferenceError: XXX is not defined」

1F12 開瀏覽器 console，看具體錯誤訊息 + stacktrace
2多半是 import 漏（重構切檔忘記補）
3找到對應檔案，加 import 即可

已發生案例：Award is not defined（Sidebar）/ SOURCE_BADGE is not defined（EmployeePermissions）

B. Migration drift（DB schema 跟 code 對不上）

典型錯誤：「column does not exist」「relation not found」

1確認哪張表哪個欄位
2到 Supabase Studio 查 information_schema.columns 確認真實 schema
3如果 schema 是新版但 migration file 是舊版 → 補 migration / 修 code
4如果是老闆直接 Studio 改 → 更新 migration file 保持同步

C. 簽核流程卡住

單據卡在某關，沒人能簽。

1查該單 approval_chain_id + current_step
2查對應 chain_step 的 target_type 跟解出來的 approver
3常見：applicant 的 supervisor / store_manager 沒設 → 解不出簽核人
4修：到員工資料補上 supervisor_id / 門市設 manager_id

D. 加簽幽靈單

workflow_instances + tasks 跟 expense_request 不同步，LINE 推「今日到期提醒」吵人。

1確認該 workflow_instance / tasks 對應的 expense_request 真實狀態
2如果原單已結案，把 wi + tasks 標已完成（不動原單）
3SQL：UPDATE workflow_instances SET status='已完成' WHERE id IN (...)

E. LINE 推不出去

檢查項方法
該員工有沒有 LINE 綁定SELECT * FROM employee_line_accounts WHERE employee_id = X
line-push Edge Function 健康看 Supabase functions log
LINE channel token 過期看 line_channels 表 access_token + 重新整理
net.http_post 失敗查 pg_net 設定 + extensions

8.2 SQL 緊急修法（要小心！）

鐵則：
  • 所有 SQL update 先 BEGIN; ... 觀察結果後再 COMMIT
  • 影響範圍要先 SELECT 列出來看清楚
  • DELETE / DROP 等不可逆動作絕對先列影響 → 確認 → 才動手
  • 2026-05-13 慘案：誤解「加班就洪伯嘉就好」結果刪錯方向，22 筆非洪伯嘉資料蒸發

第 9 章 · LINE BOT 指令清單

給 所有員工。LINE 上可以做什麼。

9.1 文字指令

指令功能範例
/註冊 姓名綁定 LINE 帳號到員工/註冊 王小明
/我的單查看自己最近的申請狀態—
/我的待簽查看輪到我簽的單—
/我的班表看本月班表（卡片）—
/薪資查最新一期薪資（卡片）—
/打卡跳 LIFF 打卡頁—
/請假跳 LIFF 請假頁—
/幫助列所有可用指令—

9.2 卡片按鈕（postback）

LINE 收到的 flex 卡上有按鈕，點下去會做：

按鈕動作
✅ 核准對該單核准（不用進 LIFF）
❌ 退回跳 quick reply 選常用原因，或進 LIFF 自己寫
📋 明細跳 LIFF 看完整資料
✏️ 編輯重送（被退回後）跳 LIFF 編輯重新送審
🪶 核准加簽對加簽請求直接核准（不用進 LIFF）

9.3 LINE 通知卡片類型

卡片什麼時候收到
簽核請求卡輪到你簽某張單
加簽請求卡有人邀請你協助加簽
已核准通知自己的單通過了
已退回通知自己的單被退回（含原因 + 編輯重送鈕）
到期提醒排程提醒當日待簽（每天早上推）
任務確認卡任務 chain 輪到你確認

9.4 LINE 卡片不見了？

LINE 客戶端有快取，如果系統 deploy 後卡片格式變了，可能要重新觸發指令才能看到新卡。

1輸入指令重新觸發（例：/我的待簽）
2或請發出單據的人重新觸發那個事件

9.5 LINE 收不到通知

檢查怎麼做
LINE 有沒有封鎖官方帳號檢查好友列表
有沒有解除綁定輸入 /我的單 看會不會回
LINE 通知總開關LINE app → 設定 → 通知
請 HR / Admin 重新綁後台檢查 employee_line_accounts
